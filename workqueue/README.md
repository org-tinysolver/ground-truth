# Workqueue

작업 관리를 위한 문서 기반 큐 시스템.

## Goal

**IssueHunt 스타일의 바운티 기반 작업 체계 구축**

누구나 작업을 등록하고, 누구나 작업을 가져가서 완료할 수 있는 구조.
작업의 목적(Why)과 범위(What)가 명확해야 다른 사람이 픽업해서 진행 가능.

### 레퍼런스

| 서비스 | 설명 |
|--------|------|
| [IssueHunt](https://issuehunt.io/) | 오픈소스 이슈 바운티 플랫폼 |
| [Algora](https://algora.io/) | 오픈소스 바운티 (GitHub 연동) |
| [Gitcoin](https://gitcoin.co/) | Web3/블록체인 기반 바운티 |
| [Open Collective](https://opencollective.com/) | 오픈소스 펀딩/후원 플랫폼 |

### 왜 이런 구조인가?

1. **작업 독립성**: 한 파일 = 한 작업. 담당자가 바뀌어도 컨텍스트 유실 없음
2. **Why 중심**: 목적이 명확해야 작업 범위가 명확하고, 결과물 검증도 가능
3. **유형별 분류**: 바운티 필터링 용이 (bug만 보기, docs만 보기)
4. **상태/우선순위**: 작업 선택시 판단 기준 제공

## 구조

```
workqueue/
├── bug/        # 버그/이슈 수정
├── infra/      # 인프라/DevOps 설정
├── docs/       # 문서 작업
├── huddle/     # 미팅/공유사항
├── chore/      # 잡무/관리
└── work-template.md
```

## 작업 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| `bug` | 버그/이슈 수정 | API 에러, 권한 문제 |
| `infra` | 인프라 설정 | 배포 파이프라인, 클러스터 연동 |
| `docs` | 문서 작업 | 온보딩 가이드, API 문서 |
| `huddle` | 미팅/공유 | 기술 결정, 아키텍처 논의 |
| `chore` | 잡무/관리 | 리팩토링, 의존성 업데이트 |

## 문서 구조 (Why 중심)

각 작업 문서는 다음 구조를 따름:

```markdown
## Why (목적)
> 거시적 관점에서 왜 이 일을 해야 하는가?

## What (무엇)
> 무엇을 달성해야 하는가?

## How (방법)
> 어떻게 할 것인가?

## Why This Way (선택 이유)
> 왜 이 방법을 선택했는가?
```

### 설계 원칙

- **What → Why**: 무엇을 하는지보다 왜 하는지가 중요
- **How → Why**: 방법을 정할 때 왜 그 방법인지 근거 필요
- **가이드, 강요 아님**: 빈 섹션이 있어도 괜찮음. 작업하면서 채워나가기

## 상태 관리

| 상태 | 설명 |
|------|------|
| `대기` | 아직 시작 안함 |
| `진행중` | 현재 작업 중 |
| `블락` | 외부 요인으로 대기 |
| `완료` | 작업 완료 |

## 우선순위

| 우선순위 | 설명 |
|----------|------|
| 🔴 높음 | 즉시 처리 필요 |
| 🟡 중간 | 이번 스프린트 내 처리 |
| 🟢 낮음 | 여유있을 때 처리 |

## 새 작업 추가하기

1. 해당 유형 폴더에 새 파일 생성
2. `work-template.md` 복사해서 사용
3. Why부터 채우기 - 목적이 명확해야 작업 범위도 명확

## 완료된 작업

완료된 작업은 파일명에 `-done` 접미사를 붙이거나, 상태를 `완료`로 변경해서 보관.
삭제해도 되지만, 히스토리 참고용으로 남겨두는 것도 방법.
