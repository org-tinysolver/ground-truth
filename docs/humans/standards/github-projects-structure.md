# GitHub Projects 보드 체계

> **Source of Truth**: 조직 내 GitHub Projects 보드 구조와 용도 정의

## 배경

### 문제
- 여러 repo에 이슈가 흩어져 전체 현황 파악이 어려움
- 팀 내부 작업과 외부 공유용 작업이 구분되지 않음
- 프로젝트(TF) 단위 진행 상황 추적 어려움
- 보드 이름만 보고 역할/성격 파악이 어려움

### 해결
GitHub Projects를 활용해 이슈를 논리적으로 그룹핑하고, **접두사로 보드 유형을 명시**:
- `[TEAM]` - 팀 자율 운영 보드
- `[PRODUCT]` - 제품/서비스 장기 보드
- `[TF]` - Task Force 한시적 보드

---

## 보드 유형과 명명 규칙

### 명명 규칙

```
[유형] 보드명
```

| 접두사 | 의미 | 특징 |
|--------|------|------|
| `[TEAM]` | 팀 보드 | AI Agent 자율 운영, 팀 내부 작업 |
| `[PRODUCT]` | 제품 보드 | 장기 프로젝트, 지속적 운영 |
| `[TF]` | Task Force | 한시적, 특정 문제 해결, 워룸 형태 |
| `[HITL]` | Human-in-the-Loop | 인간 판단 필요, 블로커 이슈, 권한 확대 기반 |

### 예시

```
[TEAM] AI PM               → PM 팀 자율 운영
[TEAM] DevTeam             → 개발팀 자율 운영
[TEAM] Research            → 리서치팀 자율 운영
[PRODUCT] MerryMatch       → MerryMatch 서비스 (장기)
[TF] 인증시스템개편         → 한시적 TF (완료 시 close)
[HITL] Human Decision Queue → 인간 판단 대기열
```

## 보드 유형별 정의

### 1. [TEAM] 팀 보드

> **AI Agent가 자율적으로 동작하는 보드**

| 항목 | 설명 |
|------|------|
| 운영 주체 | 해당 팀 AI Agent |
| 가시성 | 팀 내부 (다른 팀이 볼 필요 없음) |
| 수명 | 팀 존재하는 한 영구 |
| 포함 이슈 | 팀 내부 개선, 자가 작업, `internal` 라벨 |

**왜 [TEAM]인가?**
- 각 팀은 자체적인 백로그와 개선 작업이 있음
- AI Agent가 자율적으로 우선순위 판단하고 작업 수행
- 외부 개입 없이 팀 내부에서 관리

**현재 팀 보드:**
| 보드명 | 팀 | Repo |
|--------|-----|------|
| [TEAM] AI PM | PM | ai-pm |
| [TEAM] DevTeam | 개발팀 | ai-devteam |
| [TEAM] Research | 리서치팀 | ai-research |

---

### 2. [PRODUCT] 제품 보드

> **장기적으로 운영되는 제품/서비스 보드**

| 항목 | 설명 |
|------|------|
| 운영 주체 | PM + 담당 팀 |
| 가시성 | 관련자 전체 |
| 수명 | 제품 라이프사이클 동안 (장기) |
| 포함 이슈 | 여러 repo의 해당 제품 관련 이슈 |

**왜 [PRODUCT]인가?**
- 제품은 한 번 만들어지면 계속 유지보수/발전
- 여러 팀이 협업해야 할 수 있음
- 기능 추가, 버그 수정, 개선 등 지속적 작업

**현재 제품 보드:**
| 보드명 | 제품 | 상태 |
|--------|------|------|
| [PRODUCT] MerryMatch | 크리스마스 매칭 서비스 | active |
| [PRODUCT] TinySolver.me | 포트폴리오 사이트 | 필요시 생성 |

---

### 3. [TF] Task Force 보드

> **한시적으로 특정 문제에 집중하는 보드 (워룸)**

| 항목 | 설명 |
|------|------|
| 운영 주체 | PM (총괄) + 참여 팀들 |
| 가시성 | TF 참여자 전체 |
| 수명 | 문제 해결 시까지 (한시적) |
| 포함 이슈 | 해당 문제 관련 모든 이슈 |

**왜 [TF]인가?**
- 작업이 너무 커서 여러 팀이 동시에 봐야 함
- 긴급한 문제 해결을 위한 집중 모드
- 목표 달성 시 보드 close

**[TF] 생성 조건:**
1. 여러 팀이 동시에 참여해야 함
2. 명확한 목표와 종료 조건이 있음
3. 일반 작업 흐름으로는 해결이 어려움

**예시:**
```
[TF] 인증시스템개편      → 3개 팀 참여, 2주 집중
[TF] 성능최적화          → 전사 긴급 대응
[TF] 보안취약점대응      → 워룸 형태 운영
```

---

### 4. [HITL] Human Decision Queue

> **인간이 판단해야 하는 블로커 이슈들의 대기열**

| 항목 | 설명 |
|------|------|
| 운영 주체 | 인간 (PM/리더) |
| 가시성 | 인간 전용 |
| 수명 | 영구 (시스템 운영 동안 계속) |
| 포함 이슈 | `hitl` 라벨 이슈, 블로커, 권한 확대 필요 건 |

**왜 [HITL]인가?**

이 보드는 **AI Agent 자율성 확대의 핵심 피드백 루프**:

```
[HITL] 보드 이슈 해결
        ↓
  새로운 기준/규칙 정립
        ↓
  AI Agent 권한 확대
        ↓
  더 많은 자율 위임 가능
        ↓
  인간은 더 높은 레벨에 집중
```

**포함되는 이슈 유형:**
1. **배포 승인** - 프로덕션 배포 전 검토
2. **인프라 변경** - DNS, DB, 보안 설정
3. **정책 결정** - 새로운 규칙/기준 필요
4. **예외 처리** - 기존 규칙으로 처리 불가
5. **권한 요청** - AI가 새로운 권한 필요

**이슈 해결의 결과:**
- 해결된 이슈 → 새로운 규칙으로 `agent-docs/rules/`에 문서화
- 반복되는 패턴 → AI Agent에게 자율 처리 권한 위임
- 팀은 피드백을 통해 내부적으로 발전

**현재 HITL 보드:**
| 보드명 | 번호 | 용도 |
|--------|------|------|
| [HITL] Human Decision Queue | #5 | 인간 판단 대기열 |

## 보드 생성/종료 기준

### [TEAM] 보드
| 시점 | 액션 |
|------|------|
| 팀 생성 | 보드 자동 생성 |
| 팀 해체 | 보드 archive |

### [PRODUCT] 보드
| 시점 | 액션 |
|------|------|
| 프로젝트 active & 첫 이슈 발생 | 보드 생성 |
| 프로젝트 completed | 보드 유지 (유지보수용) |
| 프로젝트 archived | 보드 close |

### [TF] 보드
| 시점 | 액션 |
|------|------|
| PM이 TF 필요 판단 | 보드 생성 + 목표/종료조건 명시 |
| 목표 달성 | 보드 close + 회고 기록 |
| 일정 기간 비활성 | PM이 종료 여부 판단 |

## 이슈-보드 매핑 규칙

### 하나의 이슈는 여러 보드에 속할 수 있음

```
MerryMatch #45 (Firebase 설정)
├── [TEAM] DevTeam           (팀 작업으로서)
├── [PRODUCT] MerryMatch     (제품 추적으로서)
└── [TEAM] AI PM             (HITL이므로 Dashboard에도)
```

### 자동 매핑 규칙

| 조건 | 추가 보드 |
|------|-----------|
| 모든 이슈 | 해당 [TEAM] 보드 |
| `ready-for-*` 라벨 | [TEAM] AI PM |
| `hitl` 라벨 | [TEAM] AI PM |
| 제품 관련 이슈 | 해당 [PRODUCT] 보드 |
| TF 관련 이슈 | 해당 [TF] 보드 |

## 요약

| 접두사 | 용도 | 운영 방식 | 수명 |
|--------|------|-----------|------|
| `[TEAM]` | 팀 자율 운영 | AI Agent 자율 | 영구 |
| `[PRODUCT]` | 제품/서비스 | PM + 팀 협업 | 장기 |
| `[TF]` | 특정 문제 해결 | 워룸/집중 모드 | 한시적 |
| `[HITL]` | 인간 판단 대기열 | 인간 전용 | 영구 |
