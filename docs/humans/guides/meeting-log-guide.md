## 왜 회의록을 남기는가

### 문제점
- AI 에이전트와의 대화는 세션 종료 시 휘발됨
- 의사결정 과정과 맥락이 사라짐
- 같은 논의를 반복하게 됨
- 팀원(미래의 나 포함)이 결정 이유를 알 수 없음

### 해결
- 모든 중요 세션을 스냅샷으로 저장
- 의사결정의 WHY를 기록
- 나중에 참조 가능한 지식 베이스 구축

## 회의록 구조

### 필수 섹션

```markdown
# [Session Type] 세션 제목

## Metadata
- **Date**: YYYY-MM-DD
- **Duration**: ~30min
- **Session ID**: YYYYMMDD-{type}-{seq}
- **Participants**: 참여자 목록

## Purpose
이 세션의 목적과 해결하려는 질문

## Discussion
핵심 논의 내용

## Conclusions
도출된 결론과 의사결정

## Action Items
후속 작업 목록
```

### 선택 섹션

```markdown
## References
관련 문서, 링크, 이전 회의록

## Appendix
상세 데이터, 코드 스니펫, 다이어그램
```

## 작성 가이드라인

### 1. 간결하게 핵심만

❌ **나쁜 예시**
```
User가 React를 쓸지 Vue를 쓸지 물어봤고, AI가 여러 옵션에 대해 
설명을 했는데, React는 이렇고 저렇고... Vue는 이런 장점이 있고...
```

✅ **좋은 예시**
```
### React vs Vue 선택
- **결론**: React 선택
- **이유**: 기존 경험, shadcn/ui 에코시스템
```

### 2. 의사결정 중심

모든 논의가 아닌, **결론에 영향을 준 핵심 논점**만 기록

### 3. WHY를 기록

```markdown
## Decision: PostgreSQL 선택

**선택**: PostgreSQL + Prisma
**대안**: MongoDB, Supabase
**이유**: 
- 관계형 데이터 구조 적합
- Prisma의 타입 안전성
- Cloudflare D1보다 복잡한 쿼리 지원
```

### 4. 액션 아이템은 구체적으로

❌ `- 다음에 확인하기`  
✅ `- [ ] @user: Cloudflare D1 벤치마크 리서치 (Due: 2025-01-20)`

## 네이밍 컨벤션

### 폴더 구조

```
meetings/
├── 2025-01/
│   ├── 2025-01-15-DEC-001.md    # 첫 번째 의사결정
│   ├── 2025-01-15-RSC-001.md    # 같은 날 리서치
│   └── 2025-01-20-PLN-001.md    # 계획 회의
├── 2025-02/
│   └── ...
└── README.md                     # 회의록 인덱스
```

### 파일명 형식

```
YYYY-MM-DD-{TYPE}-{SEQ}.md
```

| 요소 | 설명 | 예시 |
|------|------|------|
| `YYYY-MM-DD` | 날짜 | 2025-01-15 |
| `TYPE` | 세션 타입 코드 | DEC, RSC, PLN |
| `SEQ` | 당일 순번 | 001, 002 |

### Session ID

회의록 내에서 사용하는 고유 ID:

```
20250115-DEC-001
```

## 발언 기록 형식

### 참여자 표기

| 참여자 | 표기 | 설명 |
|--------|------|------|
| 사용자 | `@user` | 인간 참여자 |
| Claude (Cursor) | `@cursor` | Cursor IDE의 Claude |
| Claude Code | `@claude-code` | Claude Code Agent |
| ChatGPT | `@chatgpt` | OpenAI ChatGPT |
| Custom Agent | `@{agent-name}` | 커스텀 에이전트 |

### 발언 기록 스타일

#### 스타일 1: 인용 블록 (권장)

```markdown
> **@user**: 프론트엔드 프레임워크 뭐 쓸까?

> **@cursor**: React + Remix 조합을 추천드립니다. 이유는...
```

#### 스타일 2: 대화 목록 (간략)

```markdown
- **@user**: 프론트엔드 프레임워크 뭐 쓸까?
- **@cursor**: React + Remix 추천 (기존 경험, SSR 지원)
```

#### 스타일 3: 요약 (핵심만)

```markdown
### 프론트엔드 선택 논의
**@user** ↔ **@cursor**

- 검토 옵션: React, Vue, Svelte
- **결론**: React + Remix
- **근거**: 기존 경험, shadcn/ui 호환, SSR 지원
```

## AI 에이전트 행동 규칙

### 1. 세션 시작 시

세션 목적을 명확히 하고 시작:

```markdown
## 세션 시작

**목적**: [사용자 요청 또는 자동 감지된 목적]
**예상 산출물**: [회의록, 코드, 문서 등]
```

### 2. 회의록 작성 트리거

다음 상황에서 회의록 작성을 **자동 제안**:

| 트리거 | 설명 |
|--------|------|
| 의사결정 완료 | 기술 스택, 아키텍처, 방향성 결정 |
| 리서치 완료 | 조사 결과 도출 |
| 긴 세션 | 30분 이상 논의 |
| 명시적 요청 | "기록해줘", "남겨줘", "회의록" |
| 중요 변경 | 기존 결정 번복, 큰 방향 전환 |

### 3. 회의록 작성 프롬프트

세션 종료 또는 트리거 발생 시:

```markdown
---
📝 **회의록 작성 제안**

이 세션에서 중요한 의사결정이 있었습니다.
회의록으로 남길까요?

- **세션 타입**: [추천 타입]
- **주요 내용**: [요약]

> 회의록으로 남기려면 "기록해줘" 또는 "회의록 작성"이라고 말씀해주세요.
---
```

### 4. 회의록 자동 생성

사용자가 동의하면:

1. 템플릿 기반으로 회의록 초안 작성
2. `meetings/YYYY-MM/` 경로에 저장
3. HISTORY.md에 링크 추가 (선택)

### 5. 이전 회의록 참조

관련 주제 논의 시 이전 회의록 참조:

```markdown
💡 관련 이전 회의록이 있습니다:
- [2025-01-10-DEC-001](meetings/2025-01/2025-01-10-DEC-001.md): 프론트엔드 기술 스택 결정
```

회의록 작성 시 `agent-docs/templates/meeting-log.md` 템플릿 사용.

2. 메타데이터 채우기

## 베스트 프랙티스

1. **즉시 작성**: 세션 직후 작성 (기억이 선명할 때)
2. **핵심만**: 모든 대화가 아닌 의사결정 포인트만
3. **맥락 포함**: WHY를 반드시 기록
4. **액션 연결**: 후속 Task/Issue와 연결
5. **정기 리뷰**: 월 1회 이전 회의록 정리
