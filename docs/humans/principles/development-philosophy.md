# Development Philosophy

> **Source of Truth**: org-tinysolver의 개발 철학과 핵심 가치관

## 핵심 가치

### 1. 점진적 개발 (Incremental Development)

> **"완벽한 계획보다 작동하는 결과물을 먼저"**

- 항상 단계별로 작업
- 큰 작업은 관리 가능한 청크로 분해
- 다음 단계로 넘어가기 전에 검증

**왜 이렇게 하는가?**
- 조기 피드백으로 방향 수정 가능
- 리스크 분산
- 진행 상황 측정 가능

### 2. MVP 우선 (MVP First)

> **"필수 기능만으로 먼저 출시"**

- 핵심 가치를 전달하는 최소 기능만 구현
- Feature creep 방지
- 사용자 피드백 기반 확장

**왜 이렇게 하는가?**
- 완벽한 제품은 없음
- 빠른 시장 검증
- 리소스 효율성

### 3. 반복적 개선 (Iterative Enhancement)

> **"출시 후에도 계속 개선"**

- 코어 출시 후 부가 기능 추가
- 피드백 기반 우선순위 조정
- 사용자 임팩트 중심 결정

**왜 이렇게 하는가?**
- 실사용 데이터가 가장 정확
- 빠른 학습 사이클
- 유연한 대응

## 우선순위 결정 기준

### Priority Levels

| Level | 이름 | 기준 |
|-------|------|------|
| P0 | Critical | 프로덕션 다운, 보안 이슈 |
| P1 | High | 핵심 기능 장애 |
| P2 | Medium | 기능 개발, 개선 |
| P3 | Low | Nice-to-have, 기술 부채 |

### 판단 요소

1. **비즈니스 임팩트**: 사용자/수익에 미치는 영향?
2. **긴급도**: 마감이 있는가?
3. **의존성**: 다른 작업이 블로킹되는가?
4. **노력도**: 구현 비용?
5. **리스크**: 뭐가 잘못될 수 있는가?

## Human-in-the-Loop (HITL)

> **AI가 자동화해도 인간의 판단은 필수**

### 왜 HITL인가?

- AI는 도구, 최종 결정은 인간
- 되돌리기 어려운 작업은 사전 승인 필요
- 책임과 통제의 균형

### HITL 필수 시점

1. **배포 전**
   - 프로덕션 배포
   - DNS 변경
   - API Breaking Change

2. **중요 변경 전**
   - DB 스키마 변경
   - 인프라 수정
   - 보안 관련 업데이트

3. **완료 후**
   - 최종 리뷰
   - 요약 확인

### HITL을 건너뛰면 안 되는 이유

```
❌ "빠르게 하려고 HITL 스킵" → 롤백 비용 >> 검토 비용
✅ "잠깐 멈추고 확인" → 안전한 진행
```

## AI Agent와의 협업 원칙

### 역할 분리

| 역할 | 담당 | 예시 |
|------|------|------|
| Human | 전략, 의사결정, 승인 | "무엇을 만들 것인가" |
| AI PM | 계획, 조율, 추적 | "어떤 순서로, 어떤 Task로" |
| Dev Agent | 구현, 실행 | "어떻게 코드로 구현" |

### 협업 규칙

1. **명확한 요청**: 모호하지 않게 요구사항 전달
2. **피드백 루프**: 결과 확인 후 방향 수정
3. **기록 유지**: 중요 논의는 회의록으로 남김
4. **신뢰하되 검증**: AI 결과물도 검토

## 기술 선택 철학

### 선택 기준

| 기준 | 우선순위 | 설명 |
|------|----------|------|
| **실용성** | 높음 | 문제를 해결하는가? |
| **친숙도** | 높음 | 팀이 잘 아는 기술인가? |
| **유지보수** | 높음 | 장기적으로 관리 가능한가? |
| **생태계** | 중간 | 커뮤니티, 라이브러리 지원? |
| **최신성** | 낮음 | 새롭다고 좋은 것 아님 |

### 현재 기술 스택 선택 이유

| 기술 | 선택 이유 |
|------|----------|
| React + Remix | SSR 지원, 파일 기반 라우팅, React 생태계 |
| TypeScript | 타입 안전성, 자동 완성, 리팩토링 용이 |
| TailwindCSS | 빠른 스타일링, 일관된 디자인 시스템 |
| PostgreSQL + Prisma | 관계형 데이터, 타입 안전 ORM |
| Cloudflare | 글로벌 CDN, 저렴한 비용, 간편한 배포 |

## 의사결정 프레임워크

### 새 기술 도입 시

1. **문제 정의**: 현재 무엇이 부족한가?
2. **대안 검토**: 최소 2-3개 옵션 비교
3. **POC**: 작은 규모로 검증
4. **결정 & 기록**: 선택 이유를 ADR로 남김
5. **점진적 도입**: 한 번에 다 바꾸지 않음

### 질문 체크리스트

- [ ] 이게 정말 필요한가?
- [ ] 기존 것으로 해결 안 되는가?
- [ ] 팀이 배울 수 있는가?
- [ ] 유지보수 비용은?
- [ ] 롤백 가능한가?

## 실패에서 배우기

### 실패 대응 원칙

1. **빠른 인지**: 문제를 빨리 발견
2. **투명한 공유**: 숨기지 않고 기록
3. **원인 분석**: 왜 발생했는가?
4. **재발 방지**: 시스템으로 막을 수 있는가?
5. **비난 금지**: 사람이 아닌 프로세스 개선

### 회고 습관

- 분기마다 회고 (retrospective.md)
- 성과뿐 아니라 실패도 기록
- "다음에 뭘 다르게 할까?" 질문

## 요약: 핵심 원칙 5가지

1. **점진적으로** - 작은 단계로 나눠서
2. **MVP 먼저** - 핵심부터 출시
3. **기록으로** - 결정과 이유를 남기기
4. **인간 최종** - HITL 체크포인트 유지
5. **배움 지속** - 실패에서 개선
