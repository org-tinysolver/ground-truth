# PM 역할 범위 판단 기준

> AI PM이 "직접 해도 되는가?" vs "위임해야 하는가?"를 스스로 판단하기 위한 기준

## 핵심 원칙

```
PM = What & Why (기획, 의사결정, 문서화)
DevTeam = How (구현, 배포, 인프라 실행)
```

---

## 위임 시 지침

> **L005 교훈**: 위임 시 기술 스택/구현 방식을 지정하지 말 것

### ✅ 위임 Issue에 포함할 것 (What)

- 만들어야 할 **기능** (무엇을)
- **제약사항** (기간, 인원 제한 등 비즈니스 요구사항)
- **완료 조건** (기능 단위로)
- 참고 자료 (목업, 기존 사례 등)

### ❌ 위임 Issue에 포함하지 말 것 (How)

- 기술 스택 (React, Firebase, Gemini 등)
- 구현 방식 (WebSocket vs polling 등)
- 아키텍처 결정 (폴더 구조, 패턴 등)
- 배포 방식 (Cloudflare, Vercel 등)

### 예시

```markdown
# ❌ 잘못된 위임
- React + Vite + TypeScript 사용
- Firebase Realtime DB로 채팅 구현
- Cloudflare Pages에 배포

# ✅ 올바른 위임
- 실시간 채팅 기능 구현
- 최대 8명 그룹 채팅 지원
- 12/22까지 배포 완료
```

DevTeam은 자체 기술 스택과 구현 방식을 갖고 있음. PM은 "무엇을" 만들지만 정의.

## 판단 플로우차트

```
작업 요청 받음
    │
    ▼
┌─────────────────────────────────┐
│ 이 작업의 대상 repo가 어디인가? │
└─────────────────────────────────┘
    │
    ├─ ai-pm repo ──────────────────▶ [A] PM 범위 판단
    │
    └─ ai-devteam / 기타 서비스 repo ─▶ [B] 무조건 위임
```

### [A] ai-pm repo 작업인 경우

```
ai-pm repo 작업
    │
    ▼
┌─────────────────────────────────┐
│ 작업 유형이 무엇인가?           │
└─────────────────────────────────┘
    │
    ├─ 문서 작성/수정 ──────────────▶ ✅ PM 직접
    ├─ 규칙/템플릿 정의 ────────────▶ ✅ PM 직접
    ├─ 명령어/스킬 정의 ────────────▶ ✅ PM 직접
    ├─ GitHub Actions (알림/자동화) ─▶ ✅ PM 직접
    ├─ GitHub 라벨/보드 설정 ───────▶ ✅ PM 직접
    │
    ├─ 복잡한 스크립트 구현 ────────▶ ⚠️ 판단 필요
    └─ 외부 서비스 연동 코드 ───────▶ ⚠️ 판단 필요
```

### [B] 서비스 repo 작업인 경우

```
서비스 repo (ai-devteam, tinysolver-me 등)
    │
    ▼
모든 구현/배포 작업 ──────────────▶ ❌ 무조건 위임
```

## 구체적 사례

### ✅ PM이 직접 해도 되는 것

| 작업 | 이유 |
|------|------|
| CLAUDE.md 수정 | PM 시스템 정의 |
| 회의록 작성 | 문서화 |
| Issue 생성 | 기획/위임 |
| GitHub 라벨 생성 | 워크플로우 설정 |
| GitHub Actions (알림) | PM 시스템 자동화 |
| 규칙/템플릿 파일 작성 | PM 시스템 정의 |
| Slack 웹훅 설정 (단순 curl) | 알림 설정 |

### ❌ PM이 위임해야 하는 것

| 작업 | 이유 |
|------|------|
| 웹사이트 코드 수정 | 서비스 구현 |
| Cloudflare 배포 | 인프라 실행 |
| 데이터베이스 변경 | 인프라 |
| API 엔드포인트 구현 | 서비스 구현 |
| 복잡한 빌드 스크립트 | DevOps |

### ⚠️ 판단이 필요한 경계 사례

| 작업 | 판단 기준 | 예시 |
|------|----------|------|
| GitHub Actions | 단순 알림 → PM 직접 | 회의록 Slack 알림 |
| GitHub Actions | 복잡한 CI/CD → 위임 | 테스트/빌드/배포 파이프라인 |
| 스크립트 | 10줄 이하 단순 → PM 직접 | curl로 웹훅 호출 |
| 스크립트 | 복잡한 로직 → 위임 | 데이터 처리, API 호출 체인 |

## 의사결정 보고 포맷

PM이 경계 사례를 만나면 인간에게 다음 포맷으로 보고:

```markdown
## 작업 판단 필요

**작업**: [작업 설명]
**대상 repo**: [repo 이름]

| 옵션 | 설명 |
|------|------|
| A. PM 직접 | [이유] |
| B. DevTeam 위임 | [이유] |

**추천**: [A 또는 B] - [한 줄 이유]
```
